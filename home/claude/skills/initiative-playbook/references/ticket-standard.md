# Engineer-facing ticket standard

The bar every story and spike is rewritten to in phase 5. Written for **expert engineers**: give them the goal, grounded context, constraints, and acceptance — then trust them on the *how*.

## Structure

Lead with the work, never the process. Use these sections; **omit any that are empty**:

| Section | Content |
|---|---|
| `## Goal` | 1 to 2 lines — the outcome this ticket delivers. |
| `## What exists today` | Grounded reality: real file paths, component names, field names, env vars, current behavior. This is what makes the ticket trustworthy — it proves the plan was checked against the code. |
| `## Scope` | Outcome bullets — what must be true when done. Not a step-by-step recipe. |
| `## Out of scope` | Work delegated elsewhere, named **by tracker key** (e.g. "redirects handled in PROJ-1340"). |
| `## Dependencies` | Blocking tracker keys + named external inputs (an account, a credential, a sign-off). |
| `## Open inputs` | Only *real* unknowns, each named with who owns the answer. Empty section → omit it. |
| `## Acceptance` | Given/When/Then criteria. Testable. |

## Never in a ticket

- Splitting-pattern names (e.g. "workflow steps split", "SPIDR").
- Planning dates or sprint/sequence labels.
- Provenance notes: "found while breaking down", "corrected after review".
- Internal sequence IDs (`S5a`, `DP1`).
- Planning-doc cross-references ("see readiness-review.md").
- A methodology footer or any "generated by" trailer.
- The `--` (double dash) character in prose — use an em dash (`—`) or proper punctuation.

Reference sibling tickets **only by tracker key**.

## Example

```markdown
## Goal
Add a CSV export to the reports screen so users can pull their own data without
filing a support ticket.

## What exists today
Reports render from `src/reports/ReportTable` (data fetched in `useReportData`).
There is no export path today: users email support, who run a manual query by hand.
The table already holds the exact rows an export would need.

## Scope
- A working "Export CSV" action on the reports screen.
- Export reflects the user's currently-applied filters.
- Streams server-side so large accounts don't time out.

## Out of scope
- XLSX / PDF formats (PROJ-212).
- Scheduled / recurring exports (PROJ-240).

## Dependencies
- Object-storage bucket for generated files (named input: platform team).
- Blocks: PROJ-198 (filter state must be URL-addressable first).

## Acceptance
- Given a user with active filters, when they click Export CSV, then they receive
  a file containing exactly the filtered rows.
- Given an account with 1M+ rows, when they export, then the request streams and
  completes without a timeout.
```
